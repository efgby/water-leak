这个 AI Agent 系统为供水管网漏损分析系统提供**自动化决策支持**，包括：

- **感知模块** - 实时读取水表状态、流量记录、维修人员信息
- **决策模块** - 智能分析漏损异常，生成优先级调度方案和故障诊断
- **执行模块** - 将决策落地，更新数据库中的人员指派
- **交互模块** - 前端可视化展示调度和诊断结果

## 💡 使用流程

### **场景：发现水表漏损，需要派遣维修人员**

#### **第1步：点击「感知 & 分析」**
- 🧠 AI Agent 会：
  1. 读取所有水表的位置、使用量、漏损率
  2. 读取所有流量记录（用于诊断趋势）
  3. 读取所有维修人员的状态（待处理/处理中/已完成）
  4. **自动分析**出异常的水表（漏损率 > 12%）
  5. **智能排序**：按漏损严重程度排序
  6. **自动指派**：轮流分配空闲维修人员

#### **第2步：查看「优先维修顺序」表格**
- 表格显示：
  - **优先级** - 从1开始（最紧急）
  - **水表ID** - 哪个水表需要处理
  - **漏损率%** - 漏损严重程度
  - **严重程度** - 🔴严重 / 🟠中度 / 🟡轻度
  - **指派人员** - 分配给谁
  - **状态** - 待处理

#### **第3步：查看「故障诊断详情」**
- 点击水表行 → 弹出诊断窗口：
  - **诊断结果** - 严重程度评级
  - **趋势** - 漏损是否在加重（📈上升/📉下降）
  - **原因分析** - AI 推断的漏损原因（例如：管道泄漏、接头松动等）
  - **建议措施** - 具体的维修建议

例如：
```
原因分析: 中度漏损 - 可能有中等管道破裂或多处泄漏
建议措施: 立即进行详细检查，使用相关工具定位泄漏点，准备维修物资
```

#### **第4步：点击「⚡ 执行调度」**
- AI Agent 会：
  1. 将指派的维修人员状态改为 **"处理中"**
  2. 返回执行成功信息
  3. 前端刷新后，TAB3 (👷人员管理) 中会看到人员状态已更新

#### **第5步：维修完成后**
- 手动在 TAB3 更新维修人员状态为 **"已完成"**
- 该人员再次变为空闲，可被下次调度指派

## 📊 AI 算法说明

### **决策算法**

```
1. 感知阶段
   ├─ 读取 water_location → 获取漏损率统计
   ├─ 读取 water_record → 获取历史流量（用于诊断）
   └─ 读取 water_resident → 获取人员状态

2. 决策阶段
   ├─ 识别异常：漏损率 > 12% (LEAK_THRESHOLD)
   ├─ 排序优先级：按漏损率从高到低
   ├─ 分配人员：轮转分配空闲人员给异常水表
   └─ 生成诊断：分析每个水表的漏损原因和建议

3. 诊断逻辑
   ├─ 漏损率 0% → 正常 (正常)
   ├─ 0-12% → 正常范围 (正常)
   ├─ 12-18% → 轻度漏损 (🟡轻度)
   │   └─ 原因：小管道泄漏或接头松动
   ├─ 18-36% → 中度漏损 (🟠中度)
   │   └─ 原因：中等管道破裂或多处泄漏
   └─ >36% → 严重漏损 (🔴严重)
       └─ 原因：大面积管道破裂

4. 执行阶段
   ├─ 更新 water_resident 表
   │   └─ processing_status = '处理中'
   └─ 返回执行确认

5. 交互阶段
   ├─ 前端显示调度表和诊断
   └─ 用户可手动调整或点击执行
```

### **严重程度分类**

根据漏损率（Ra）和国标 GB 17446-2012 标准：

| 漏损率 | 分类 | 颜色 | 处理优先级 | 建议措施 |
|--------|------|------|-----------|---------|
| < 12% | 正常 | 🟢绿 | - | 继续监测 |
| 12-18% | 轻度 | 🟡黄 | 低 | 定期巡检、听诊定位 |
| 18-36% | 中度 | 🟠橙 | 中 | 立即检查、准备维修 |
| > 36% | 严重 | 🔴红 | 高 | 紧急处理、可能隔离 |

## 🔧 API 端点详解

### **POST /agent/perceive**
触发感知模块，读取最新数据
```python
Response:
{
  "status": "success",
  "timestamp": "2026-05-19T20:00:11.096959",
  "locations_count": 50,
  "records_count": 600,
  "residents_count": 5
}
```

### **POST /agent/decide**
触发决策模块（需先调用 perceive）
```python
Response:
{
  "status": "success",
  "abnormal_meters_count": 8,
  "free_residents_count": 3,
  "schedule_count": 8,
  "diagnostics_count": 50
}
```

### **GET /agent/schedule?limit=10**
获取优先级调度
```python
Response:
{
  "schedule": [
    {
      "priority": 1,
      "water_id": 13,
      "severity": 20.63,
      "assigned_resident": "赵工",
      "assigned_resident_id": 4,
      "status": "待处理"
    },
    ...
  ]
}
```

### **GET /agent/diagnosis/{water_id}**
获取单个水表的诊断
```python
Response:
{
  "water_id": 13,
  "current_leakage_rate": 20.63,
  "severity_level": "中度",
  "reason": "中度漏损 - 可能有中等管道破裂或多处泄漏",
  "advice": "立即进行详细检查，使用相关工具定位泄漏点",
  "trend": "稳定"
}
```

### **GET /agent/report**
获取完整报告（前端仪表板用）
```python
Response:
{
  "timestamp": "2026-05-19T20:00:11.096959",
  "state": { ... },
  "schedule": [ ... (前10个) ... ],
  "system_summary": {
    "total_meters": 50,
    "abnormal_count": 8,
    "avg_leakage_rate": 7.46,
    "pending_tasks": 8
  }
}
```

### **POST /agent/execute?confirm=true**
执行调度决策
```python
Response:
{
  "status": "success",
  "executed_count": 8,
  "message": "成功执行 8 项调度"
}
```

## 🎯 使用建议

### **日常运维**

1. **每日检查**：早上打开系统 → 点击「感知 & 分析」查看是否有异常
2. **紧急处理**：如果发现红色异常（🔴严重），立即派遣最高优先级人员
3. **定期跟踪**：下午/晚上点击「刷新报告」查看处理进度

### **完整工作流**

```
早上: 感知 → 发现8个异常
       ↓
中午: 派遣 → 执行调度，分配人员
       ↓
下午: 跟踪 → 查看"处理中"的任务
       ↓
完成: 更新 → 在 TAB3 标记为"已完成"
       ↓
晚上: 汇总 → 运行诊断报告
```

## 🐛 故障排除

### **问题：无法连接到 API**

```
错误: ConnectionError: Failed to connect to http://127.0.0.1:8000
解决:
1. 确保 API 已启动: python api.py
2. 检查防火墙是否阻止 8000 端口
3. 检查 config.py 中的 API_BASE_URL 是否正确
```

### **问题："感知 & 分析" 显示 0 数据**

```
错误: abnormal_meters_count: 0
解决:
1. 确保已生成演示数据（TAB5）
2. 检查数据库连接是否正常
3. 查看 MySQL 中 water_location 和 water_record 是否有数据
```

### **问题：诊断窗口加载缓慢**

```
现象: 点击诊断后等待较长时间
原因: 第一次调用 API 需要计算诊断数据
解决: 耐心等待，后续调用会缓存结果
```
